# 项目需求文档 (PRD)

## 1. 项目概况

**项目名称**：AI Tarot Yes or No

## 2. 核心目标

1. **问题引导**：用户提出初步问题，在 AI 的辅助下，提炼并确认出一个可用“肯定/否定（Yes or No）”回答的核心问题。
2. **拟真占卜**：APP 提供具有真实沉浸感的拟真塔罗占卜流程（洗牌、切牌、抽牌）。
3. **牌面解读**：AI 结合用户确定的核心问题与占卜抽取的塔罗牌结果，提供专业解读并给出明确的“肯定/否定”答案。

## 3. 用户操作流程

1. **输入问题**：用户输入初始问题。
2. **AI 引导**：AI 判断该问题是否符合要求，并通过对话与用户反复交流，不断给出近似的标准化问题提示（APP 侧以快捷聊天回复气泡的形式展示），直到引导用户确认一个可以用“肯定/否定”回答的核心问题。确认后，提示用户回复“开始占卜”。
3. **占卜流程**：APP 进入完整的视觉交互占卜流程：
   - **洗牌**：通过自然拟真的完美洗牌动画，模拟真实的洗牌过程。
   - **切牌**：通过自然拟真的切牌动画，模拟真实的切牌过程。
   - **抽牌**：通过自然拟真的抽牌动画，模拟真实的抽牌过程。
4. **结果解读**：AI 结合核心问题和抽取的塔罗牌，输出详细解读以及明确的“肯定/否定”答案。
5. **后续操作**：用户可选择继续提问或退出应用。

## 4. 页面结构设计

1. **开屏启动页**：用户选择性别，系统自动生成对应的初始头像。
2. **首页（对话界面）**：采用主流的聊天界面布局。左上角展示用户头像。用户在此与 AI 进行问题沟通，占卜结束后 AI 也在该界面输出塔罗牌解读结果。
3. **占卜页面**：通过全屏拟真动画展示洗牌、切牌、抽牌等流程，提供沉浸式的真实占卜体验。
4. **侧边栏**：点击首页左上角头像呼出，包含：历史记录、塔罗主题、塔罗牌图鉴、设置。
5. **历史聊天记录**：列表展示过往的占卜对话记录，点击可进入详情页面查看。
6. **塔罗主题**：
   - 用户可在此浏览并更换APP主题（包含整套UI色彩和对应风格的塔罗牌面）。
   - 支持主题内购功能。
   - 包含创作者生态：允许用户申请成为设计师，按规范上传自定义主题，经平台审核后上架发布。
7. **塔罗牌图鉴**：
   - 以画廊（Gallery）形式展示所有塔罗牌。中心高亮显示当前卡牌，左右两侧虚化显示上一张/下一张。
   - 支持左右滑动切换。
   - 卡片下方展示卡牌资料：名称、正位含义、逆位含义。
   - 支持将卡牌信息分享为图片或链接，并可点击查看更多详细信息。
8. **设置中心**：包含账号管理、通知设置、隐私政策、用户协议、关于我们等常规功能。

## 5. 视觉与交互设计

### 5.1 视觉风格

1. **整体原则**：界面规范、精简不冗余；色彩搭配统一，对齐与间距克制且一致。动效自然连贯，无突兀感；无多余元素与繁琐操作，无元素错位与遮挡，杜绝突兀弹窗，尽量减少页面层级的跳转切换。
2. **适配要求**：
   - 窄屏设备（移动端）**不需要侧边栏**，完全收起或通过其它入口导航。
   - 在宽屏设备（Pad/PC端）采用**分栏显示**。建议采用**三栏布局**：左侧为菜单侧边栏、中间为核心聊天区、右侧为聊天历史记录，以保证UI主体比例并在大屏上拥有沉浸式体验。
3. **默认主题设计**：**复古新艺术运动（Art Nouveau）+ 中世纪手抄本风格**
   - **整体质感**：采用做旧泛黄的牛皮纸（Kraft paper）基底，边缘附带磨损与做旧纹理，结合极具拟物感的油蜡（Oil wax）涂层光泽与黄铜（Brass）立体边框装饰，营造出古老、厚重的神秘历史感。
   - **色彩系统**：以暖棕、米黄主底色，点缀墨绿、酒红、黄铜色等低饱和度复古色调。色彩沉稳典雅，注重物理材质的渐变光影与高光反射，确保整体色调和谐统一。
4. **主题系统规范（静图覆盖式设计）**：遵循 APP 统一的设计系统。为了保证深度的拟物质感，核心 UI 控件（包括顶部的进度星星、底部的厚重按钮背景等）**必须全部采用静态图片资源（Assets）进行预渲染和加载**。切换后续不同主题时，系统会自动切包切换：全套 UI 色彩变量、按键底图、导航图标、四角星标记以及塔罗牌正反面设计图，实现最深度的重绘体验。
5. **占卜页面遮罩**：占卜时通过毛玻璃蒙版（Glassmorphism）覆盖在主聊天界面上，在此层展示洗牌、切牌、抽牌动画。

### 5.2 动效设计

#### 页面切换动效

1. **聊天页 ↔ 占卜页**：无硬性跳转，通过背景模糊效果平滑过渡。占卜页悬浮于聊天页之上，聊天内容隐约可见。占卜结束后模糊层消散，APP 聊天区弹出新气泡，依次翻转展示抽到的三张卡牌面；随后显示“正在解读中”的提示，等待 AI 流式输出结果。
2. **聊天页 ↔ 侧边栏**：无页面跳转。侧边栏从左侧平滑滑出，占据屏幕约 30% 并覆盖部分聊天页。关闭时反方向滑回。
3. **其他层级跳转**：统一规范为由右侧滑入（进入新页面），由右侧滑出（返回上一页）。

#### 占卜核心动效

> **进度与阶段提示**：占卜界面上方区域使用三颗**“具有金属黄铜质感的四角星（静态图片资源）”**展示当前进度。每进入新阶段，对应星星触发点亮状态图。下方文字要求**极致精简**，仅提示用户当前行为阶段（例如：“请洗牌”、“请切牌”、“请抽取牌阵”），绝不能输出关于动画底层机理的描述词（如：“强制回正中”等）。底部的操作按钮同样需要加载出带强烈金属质感的静态图作为物理底座垫衬。

1. **洗牌（完美洗牌模拟）**：
   - **渲染要求**：屏幕中央必须是一叠具有真实厚度与层叠感的三维牌堆，切勿渲染成单张牌；分牌后同样保持左右两叠的物理厚度感。
   - **初始帧**：牌堆于屏幕正中央静止。
   - **过程**：牌堆一分为二（左右两叠），随后两侧牌堆向中央聚拢，**模拟真实发牌时纸牌的弹性与弯曲，两侧卡牌从底部开始一张张交替落下互插**，形成连贯且致密的交错视觉，最终滑入居中聚拢成全新牌堆。
   - **过程**：牌堆一分为二（左右两叠），模拟完美洗牌的过程，卡牌交替重叠，最终聚拢成一叠。
   - **结束帧**：牌堆复原于屏幕正中央。
2. **切牌**：
   - **自适应排列逻辑**：系统需首先判断当前屏幕尺寸。如果屏幕尺寸足够（宽屏/横屏），三叠牌按**左、中、右**水平排列分布；如果屏幕空间不足（窄屏/竖屏），则按**上、中、下**垂直排列分布。
   - **初始帧**：牌堆于屏幕正中央静止。
   - **过程**：牌堆平分为三叠，卡牌通过一系列动画最终到达指定的左中右（或上中下）位置，然后指定位置的两叠进行交换，最后重新叠放聚拢为一叠。
   - **结束帧**：牌堆复原于屏幕正中央。
3. **抽牌与展示**：
   - **自适应排列逻辑**：与切牌同理，宽屏为横向左中右排布，窄屏为纵向上中下排布。
   - **初始帧**：牌堆于屏幕正中央静止。
   - **过程**：
     - **准备阶段**：牌堆轻微晃动，伴随轻微屏幕震动反馈（视设备支持情况）。
     - **第一张牌落下**：首张牌从牌堆脱落。视口焦点跟随卡牌下移，**抽取时不上翻（背面朝上）**，停靠时可且带有一定的自然倾斜。
     - **溺水上浮效果（一/二/三）**：三张牌依次执行下沉再上浮停靠动作（呈现“先快后急缓”的视觉阻滞感）。
     - **阵型整理与对齐**：三张牌就位后，必须有一段专属的整理动画，让所有微微倾斜的卡牌**调整成完全垂直（0度旋转），严格排成一条直线，并且相互之间留有均匀间距**。
     - **放置桌面展开**：排列对齐完成后，整体执行一个 **Z轴（3D空间）平放至桌面** 的动效，模拟将牌铺在神坛上的纵深感。
     - **依次翻牌**：放置到桌面后，三张牌**依次**呈现被揭开的翻转效果，展示正面神谕牌面。

#### 其他微交互动效

1. **快捷回复气泡**：出现于底部输入框上方，动画为从右至左滑入，并在终点带有轻微弹簧回弹（Spring）效果，速度曲线先快后慢。
2. **边界回弹**：所有可滑动区域（如列表、画廊）触碰边界时，提供弹性阻尼反馈及回弹动画。
3. **AI 回复输出**：文本采用打字机（Typewriter）流式输出效果。
4. **AI 思考状态**：展示为一颗旋转的水晶球，球体表面伴有星星光芒随机闪烁动画。

## 6. 技术栈选型

### 6.1 前端技术栈

1. **应用框架**：uni-app
2. **开发框架**：Vue 3
3. **开发语言**：TypeScript
4. **构建工具**：Vite
5. **状态管理**：Pinia
6. **样式框架**：Tailwind CSS
7. **包管理**：yarn

### 6.2 后端技术栈

1. **开发/测试环境**：Vite local server 代理及模拟
2. **生产环境**：
   - **Web 端**：Python (如 FastAPI/Flask)
   - **微信小程序端**：微信云函数
   - **iOS 端**：Python
   - **Android 端**：Python
   - *注：大模型能力接入统一由对应的服务端处理并暴露接口。*

## 7. 版本规划 (Roadmap)

- **v1.0**：实现 Web 端，确保全套核心流程完整可用。
- **v1.1**：移植/发布微信小程序端，确保核心功能完整一致。
- **v2.0**：发布 iOS 客户端。
- **v3.0**：发布 Android 客户端，完成全平台覆盖。

## 8. 测试规范

1. **全流程测试**：覆盖正反向链路，确保占卜引导及后续解读闭环。
2. **性能测试**：重点关注复杂动效（洗牌、抽牌）在不同端/低端机型上的帧率表现。
3. **UI动效测试**：确保所有UI动效符合设计稿要求。
4. **兼容性测试**：验证多端UI适配及跨平台接口一致性。
5. **安全性测试**：保障用户数据及请求的安全性。
6. **可用性测试**：优化交互体验，验证产品可用性。
7. **自动化测试**：对核心逻辑实现自动化用例覆盖。

## 10. 项目代码结构

**核心要求**：目录精简、结构清晰、易于维护；严禁重复功能模块与冗余文件。

```text
app/                      # 项目主工程目录
├── src/                  # 前端源代码目录
│   ├── components/       # 全局复用组件目录
│   ├── pages/            # 项目页面目录
│   ├── store/            # 状态管理目录 (Pinia)
│   ├── utils/            # 通用工具函数目录
│   ├── App.vue           # 根组件/项目入口
│   └── main.ts           # 前端应用入口文件
├── static/               # 项目静态资源目录
├── dist/                 # 生产打包目录
├── package.json          # Node 依赖配置
├── README.md             # 项目说明文档
└── server/               # 后端及服务端目录
    ├── dev_server/       # 开发服务器
    ├── prod_server/      # 生产服务器
    ├── wx_server/        # 微信小程序后端 (对接云函数及大模型)
    ├── app_server/       # Web/iOS/Android 端统用服务端及大模型对接
    ├── sql/              # 数据库相关脚本
    └── LLM_config/       # AI 大模型配置及 Prompt 设计
```
